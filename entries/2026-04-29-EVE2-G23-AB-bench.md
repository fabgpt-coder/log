# G23 config-keys + A/B model bench

The morning's PR #8 audit (closed) revealed a quality gap that the
morning critic stack didn't cover: gemma-4-e4b emitted a syntactically
valid pyproject.toml whose `[tool.X]` sections used INVENTED or
MISSPELLED keys (`suppress_missing_imports` should be
`ignore_missing_imports`; `[tool.coverage]` with made-up keys like
`ignore_patterns` and `run_if_covered`). G10 (semantic config schema
validator) covers JSON-shaped configs but not pyproject TOML tool
tables — each tool has its own valid-key set. G23 fills the gap.

## What G23 does

Two-tier validator on every touched `pyproject.toml`:

1. **Typo dictionary** — high-confidence misspellings mapped to the
   correct key. The bad value is rejected with an explicit
   "use this instead" suggestion in the LLM feedback.
2. **Closed-set known-keys** — for the catalog tools (mypy, ruff
   + sub-tables, pytest.ini_options, coverage.{run,report,paths,
   html,xml}), unknown keys are flagged as "not in documented schema".

Conservative passthrough for tools we don't catalog (poetry,
setuptools, hatch, pyright, custom). False-negative bias by design.

Diff-aware: when worker provides the pre-patch text, only NEW keys
get flagged — pre-existing typos in the baseline are left alone.

Opt-in via `GITOMA_G23_CONFIG_KEYS=1` (default OFF). Same revert+retry
shape as G20-22.

32 tests. Suite 1949 → 1981.

## Live-fire validation — same target, G23 ON vs OFF

The same target that produced PR #8's bad config got re-run on the
same model (gemma-4-e4b) with the same env minus G23, then with G23
on:

| Output dimension | PR #8 (no G23) | PR #9 (G23 ON) |
|---|---|---|
| `[tool.mypy]` keys | invented (`suppress_missing_imports`) | ✓ correct (`ignore_missing_imports`) |
| `[tool.coverage]` section | invented keys | not emitted (G23 forced retreat) |
| Diff size | 156 lines | 49 lines |
| Subtasks done | 6/8 | 3/7 |
| T001-S01 (ruff) | shipped (with bad keys) | failed cleanly after 2 G23 attempts |

The "less but clean" pattern. The model couldn't ship valid ruff
config in 2 retries → G23 correctly failed the subtask → no garbage
ruff config landed on the branch. Worker still completed the OTHER
subtasks (mypy with clean keys, governance docs).

## A/B — gemma-4-e4b vs qwen3-8b on the same target, G23 ON

Same target, same critic stack, two worker models on two endpoints
(parallelism restored — see endpoint note below).

| Quality dimension | gemma-4-e4b PR #9 | qwen3-8b PR #10 |
|---|---|---|
| Tasks done | 1/4 | 2/4 |
| Subtasks | 3/7 | 3/5 |
| `[tool.mypy]` keys | clean | clean (`check_untyped_defs` + `ignore_missing_imports`) |
| `[tool.poetry]` | not emitted | weird (`dependencies = [...]` as list, lists `poetry` as own dep) |
| Critics fired | G7+G8+G23 | G7+G20+G23 (recovered via build_retry) |
| Diff size | 49 lines | 13 lines |
| Style | conservative — fails clean | aggressive — sloppy first, recovers |

Both nettamente migliori dell'iter no-G23. Different failure profiles:

- **Conservative model** ships less but cleaner. When uncertain,
  fails the subtask (which the critic stack converts into "no
  patch lands").
- **Aggressive model** ships more but generates more first-pass
  garbage. The critic stack's revert+retry mechanism does heavier
  lifting on its output.

For an unattended agent, conservative is safer (worse-case = small
PR). For a closely-supervised workflow, aggressive ships more raw
material per run.

## G24 candidate

Qwen3-8b's PR #10 has a `[tool.poetry] dependencies = [...]` block
that's structurally wrong — poetry uses a TABLE for deps
(`[tool.poetry.dependencies] foo = "^1.0"`), not a list. G23
silently passes `[tool.poetry]` because we don't catalog it
(uncatalogued = passthrough). G24 could extend coverage to packaging-
backend table SHAPES (not just keys) — backlog item.
